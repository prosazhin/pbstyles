# AGENTS.md

## Назначение

Этот документ задает единые правила для любых AI-агентов и автоматизаций, которые вносят изменения в проект `pbstyles`.

## Контекст проекта

- npm-пакет: `@prosazhin/pbstyles`
- Текущая версия: смотреть в `package.json`
- Основная цель: генерация и поставка базовых стилей/токенов (`styles/*`)
- Сборка: `npm run build`

## Базовые правила изменений

1. Не делать разрушительных git-операций (`reset --hard`, принудительные откаты), если это явно не запрошено.
2. Не менять поведение релизного процесса без явной задачи на это.
3. Любые изменения в `package.json` (имя, версия, scripts, publishConfig) синхронизировать с `package-lock.json`.
4. После изменения зависимостей обновлять lockfile и проверять `npm audit`.
5. Не редактировать `node_modules`.

## Что проверять перед завершением задачи

Минимальный чек:

1. `npm run build`
2. `git diff --name-only` и ручная проверка измененных файлов
3. Если менялись зависимости: `npm audit`

Если команда не может быть выполнена (например, нет сети), это нужно явно отметить в результате.

## Правила для релизных изменений

Файлы релиза:

- `.github/workflows/prepare.yml`
- `.github/workflows/release.yml`
- `package.json`

При изменениях релизного контура проверять:

1. Имя пакета везде одинаково: `@prosazhin/pbstyles`
2. `prepare.yml` сравнивает версию в `package.json` с предыдущим коммитом (`HEAD~1`)
3. Тег релиза имеет формат `@prosazhin/pbstyles@v<version>`
4. `release.yml` публикует с `--access public`
5. Используемые секреты:
   - `NPM_TOKEN`
   - `ACCESS_TOKEN`

## Ограничения по стилю правок

1. Делать узкие, целевые изменения.
2. Не выполнять массовые переименования без необходимости.
3. В документации писать конкретные шаги и команды без «воды».

## Формат результата от агента

В итоговом отчете указывать:

1. Что изменено (список файлов).
2. Почему внесены изменения.
3. Что проверено командами.
4. Что не удалось проверить (если есть) и почему.
